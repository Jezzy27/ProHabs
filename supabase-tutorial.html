1. 我們在解決什麼問題？

改造前，ProHabs 的習慣資料存在瀏覽器的 localStorage。你可以把它想像成瀏覽器裡的一個小抽屜，只屬於那台裝置上的那個瀏覽器。

問題說明
換裝置資料消失手機的抽屜 ≠ 電腦的抽屜，各存各的
清除瀏覽器資料就沒了抽屜被清空了，資料永遠不見
朋友無法使用朋友有自己的抽屜，不是你的資料
沒有登入保護任何人打開連結都能直接進入 App

改造後：資料存在雲端（Supabase），就像一個需要鑰匙才能開的雲端保險箱，任何裝置用同一個帳號登入，都能拿到同一份資料。

2. Supabase 是什麼？

Supabase 是一個「後端即服務（BaaS）」平台，免費提供你原本需要自己架設伺服器才能做到的功能。它幫你同時解決兩件事：

🔐 Auth（身份驗證）

類比：就像大樓的門禁系統，只有有感應卡的住戶才能進去，訪客不能隨便進門。

Supabase Auth 幫你管理「誰可以登入」，你完全不需要自己寫登入邏輯。它處理：

  • 帳號密碼的儲存（密碼是加密的，連你自己在後台也看不到明文）
  • 登入 / 登出 / Session 管理
  • 邀請新用戶（發邀請信）
  • Token 發放（登入成功後的通行證）

🗄️ Database（資料庫）

類比：就像 Google 試算表，但更安全、更快，而且可以用程式碼直接讀寫。

Supabase 使用 PostgreSQL 資料庫。我們為 ProHabs 建了這樣一張表：

user_data 表 ┌─────────────────────┬─────────────────────────┬──────────────────┐ │ user_id │ data │ updated_at │ ├─────────────────────┼─────────────────────────┼──────────────────┤ │ 你的帳號 ID │ 你所有的習慣資料 (JSON) │ 最後更新時間 │ │ 朋友A 的帳號 ID │ 朋友A 的習慣資料 │ 最後更新時間 │ │ 朋友B 的帳號 ID │ 朋友B 的習慣資料 │ 最後更新時間 │ └─────────────────────┴─────────────────────────┴──────────────────┘

每個人一行，資料完全獨立。

3. 今天做了哪些事？

Step 1：建立 Supabase 專案

就像去銀行開戶。建好後拿到兩個重要資訊：

項目說明類比
Project URL你的 Supabase 雲端地址銀行的地址
Anon KeyApp 連接 Supabase 的公開金鑰進銀行大廳的門卡（公開沒關係，進了還要自己的密碼才能開格子）

Step 2：建資料表 + 安全規則

在 Supabase SQL Editor 執行：

-- 建一張表，每個用戶存一行
create table user_data (
  user_id    uuid references auth.users primary key,
  data       jsonb not null default '{}',
  updated_at timestamptz default now()
);

-- 開啟 Row Level Security（每人只能看自己的資料）
alter table user_data enable row level security;

-- 定義規則：user_id 必須等於目前登入的人
create policy "own data only" on user_data
  for all using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

Step 3：改 App 程式碼

改了什麼效果
加入 Supabase JS SDKApp 可以跟 Supabase 溝通
加登入頁面進 App 前先驗證身份
加設定密碼頁面朋友收到邀請信後可以設定自己的密碼
改造 saveData()打卡時同時存本地 + 推雲端
加 syncFromCloud()開 App 時從雲端拉最新資料
設定頁加登出按鈕可以切換帳號

Step 4：關閉公開註冊

在 Supabase → Authentication → Sign In / Providers → 關掉「Allow new users to sign up」。

之後只有你在 Dashboard 手動建立或邀請的帳號才能登入，陌生人就算打開 App 也無法自己申請帳號。

Step 5：設定邀請連結的導向網址

Authentication → URL Configuration → Site URL 改成你的 GitHub Pages 網址，這樣朋友收到邀請信點連結才會正確導向 App，而不是 localhost。

4. 背後怎麼運作的？

登入流程

你輸入 Email + 密碼 ↓ Supabase 驗證帳密是否正確 ↓ ✅ 正確 → 發給你一個 Token（通行證）存在瀏覽器裡 ❌ 錯誤 → 顯示「帳號或密碼錯誤」 ↓ 下次打開 App → 自動帶上 Token → 不需要重新登入
類比：Token 就像演唱會的手環，戴著它在會場裡可以自由進出，不需要每次都出示票根。Token 有效期限約一週，過期需要重新登入。

資料同步流程

打卡時

你打勾一個習慣 ↓ ① 立刻存到 localStorage（本地，毫秒內完成） ② 非同步推送到 Supabase（雲端，背景執行，不影響操作）

開 App 時

App 啟動 ↓ 先從 localStorage 讀資料（速度快，離線也能用，畫面馬上顯示） ↓ 同時向 Supabase 請求最新資料 ↓ 雲端資料回來後 → 更新畫面（若有更新才重新渲染）

邀請朋友流程

你在 Supabase Dashboard → Invite user → 輸入朋友 Email ↓ Supabase 自動寄出邀請信給朋友 ↓ 朋友點信中的連結 → 導向 ProHabs App ↓ App 偵測到邀請 Token → 顯示「設定密碼」頁面 ↓ 朋友設定密碼 → 自動登入 → 進入 App

5. 資料安全怎麼保護的？

Row Level Security（RLS）

類比：銀行的保險箱大廳人人可以進（Anon Key），但每個格子都有獨立的鎖，只有鑰匙（你的帳號）才能開自己的格子，銀行員工也不能替你開別人的。

我們設定的規則翻譯成白話：

-- 只有「目前登入的用戶」才能存取「user_id 等於自己」的那一行
auth.uid() = user_id
情境結果
你登入，查自己的資料✅ 可以
你登入，嘗試查朋友的資料❌ 資料庫直接擋住，不回傳任何資料
沒登入，查任何人的資料❌ 全部擋住
有人拿到 Anon Key，不登入直接查❌ RLS 規則擋住

Anon Key 放在程式碼裡安全嗎？

Anon Key 是公開設計的，它只代表「這個請求來自 ProHabs App」，本身沒有任何特殊權限。真正保護資料的是 RLS 規則，就算有人拿到這把 Key，沒有你的帳號密碼，依然看不到任何資料。

類比：Anon Key 就像一棟大樓的地址，知道地址可以走進大廳，但進不了任何一間住戶的門。

6. 離線怎麼辦？

我們用的是「離線優先（Offline First）」設計策略：

狀況行為
有網路打卡存 localStorage + 推 Supabase（兩份都存）
沒網路打卡只存 localStorage（先記下來）
網路恢復自動把 localStorage 的最新狀態推上 Supabase
App 切回前景自動觸發一次同步
類比：就像 Google 文件的離線模式，沒網路繼續編輯，有網路時自動幫你上傳。你永遠不會因為沒網路而無法使用 App。

7. 整體架構總結

你的手機 / 電腦 / 任何裝置 ↕ ProHabs App (index.html) ↕ ← Supabase JS SDK（橋樑） Supabase 雲端 ├── Auth（門禁系統） │ ├── 你的帳號 │ └── 朋友的帳號 └── Database（保險箱） └── user_data 表 ├── 你的習慣資料（只有你能看） └── 朋友的習慣資料（只有朋友能看）

每個人用自己的帳號登入 → 只拿到屬於自己的資料 → 任何裝置都同步 ✅

8. 常見誤解澄清

❌ 誤解：「Supabase 會去抓我的 HTML」

很多人以為填了 Site URL 之後，Supabase 就會去「讀取」你的網頁。完全不是這樣！

實際上是反過來的：是你的 App 主動去找 Supabase，不是 Supabase 去找你的 App。

❌ 錯誤理解： Supabase ──找──→ 你的 App (GitHub Pages) ✅ 正確理解： 你的 App (GitHub Pages) ──找──→ Supabase

你的程式碼裡有這兩行，這就是 App「知道去哪裡找 Supabase」的原因：

const SUPABASE_URL = 'https://owoqbbghsjwnhvvtizqn.supabase.co'
const SUPABASE_KEY = 'sb_publishable_...'

Supabase 完全不知道你的 GitHub Pages 網址是什麼，它只是在那邊等你的 App 來連線。

那 Site URL 是幹嘛的？

Site URL 的唯一用途：決定邀請信 / 重設密碼信裡面的連結要導向哪裡。

當你發邀請給朋友，Supabase 寄出的信裡有一個連結，點了要帶朋友去哪個網址？就是 Site URL 決定的。

情境Site URL 的影響
填 localhost朋友點連結 → 導向 localhost → 手機打不開（錯誤）
填 GitHub Pages 網址朋友點連結 → 導向你的 App → 正常顯示設定密碼頁面 ✅
換到 Vercel 後更新之後寄出的信連結就會指向新的 Vercel 網址

結論：Site URL 只跟「信件連結」有關，跟資料同步完全無關。

換部署平台需要改什麼？

假設你從 GitHub Pages 換到 Vercel，你只需要：

  1. 把程式碼部署到 Vercel，拿到新的網址（例如 https://prohabs.vercel.app）
  2. 去 Supabase → Authentication → URL Configuration → 更新 Site URL
  3. 完成。資料完全不受影響，因為資料在 Supabase，不在部署平台。
類比：Supabase 是倉庫，GitHub Pages / Vercel / Netlify 是店面。你換店面地址，不影響倉庫裡的貨。只需要把新地址告訴快遞公司（更新 Site URL），讓他們知道把包裹（邀請信連結）送到新地址。

GitHub Pages 程式碼公開的問題

GitHub Pages 必須用 Public repo，代表任何人都能看到你的程式碼，包含 Supabase URL 和 Anon Key。

方案程式碼是否公開說明
GitHub Pages✅ 公開免費，但 repo 必須 public
Vercel❌ 不公開免費，可連接 private repo
Netlify❌ 不公開免費，可連接 private repo

如果介意程式碼外流，可以把 GitHub repo 改成 Private，然後用 Vercel 或 Netlify 部署。這兩個平台都能連接 private repo，免費方案也夠個人使用。

一句話總結

我們把 ProHabs 從「存在瀏覽器小抽屜」升級成「需要鑰匙才能開的雲端保險箱」，

任何裝置登入都能拿到同一份資料，沒帳號的人打開也只會看到登入頁面。

記住最重要的一點：是 App 找 Supabase，不是 Supabase 找 App。
